Executive Summary

The ResistanceZero portfolio website demonstrates strong visual design, a cohesive dark/light theme system, and well-structured content hierarchy. The site presents a professional, modern aesthetic with glassmorphism effects, thoughtful typography (Inter + JetBrains Mono), and smooth transitions. The bento-grid hero layout is particularly distinctive and effective.

However, several areas require attention: mobile navigation accessibility, inconsistent interaction patterns across 17+ articles with self-contained CSS, some WCAG contrast issues in secondary text elements, and keyboard navigation gaps. The sheer scale of the site (30+ HTML files with inline styles) creates maintainability challenges that impact UI/UX consistency.

Good — With Targeted Improvements Needed

The website scores well in visual design and content structure but loses points in accessibility, responsive edge cases, and interaction consistency. The weighted overall score of 72/100 reflects a professional site with specific, addressable gaps. With the recommended fixes, the expected score rises to 86/100.

styles.css:29–121

Medium Per-Article CSS Prefix Isolation Creates Drift Risk
Each article uses unique CSS prefixes (art-6: .rca-, art-7: .res-, art-8: .shi-, etc.) with massive inline <style> blocks. While this prevents collisions, it means visual changes (KPI card sizes, panel fonts, chart colors) must be updated in 15+ files individually. Documented in standardization files but still error-prone.
Per article-N.html inline styles, standarization/ docs
Recommendation: Extract shared Pro Mode component CSS into a pro-components.css file. Keep per-article prefixes for uniqueness but share base rules via CSS layers or shared class names.
Low CSS Specificity Conflicts With Global vs. Article Styles
The global selector [class*="-kpi-value"] in styles.css has lower specificity than per-article class selectors like .ws-kpi-value. This was previously identified and fixed with !important overrides, but the pattern indicates an architectural fragility.
styles.css (bottom section), per-article styles
Recommendation: Use CSS Cascade Layers (@layer) to establish clear priority: base > components > article-specific.
Low Inline Style Attributes in Navigation HTML
The dropdown menu items use inline style="color: #06b6d4; font-weight: 600;" and style="color: #8b5cf6;" directly in the HTML. This bypasses the theming system and these colors do not adapt to dark mode.
index.html:168–174
Recommendation: Move these color assignments to CSS classes like .dropdown-item--featured with dark mode variants.

Pass Viewport Meta Tag Present on All Pages
All audited pages include <meta name="viewport" content="width=device-width, initial-scale=1.0">.
Medium Auth Module Responsive Handling Is Minimal
The auth.js responsive CSS only hides .rz-user-email and .rz-login-text below 768px. The login modal (.rz-modal) uses max-width:90vw which is good, but there are no responsive adjustments for the dropdown positioning on small screens. The dropdown could overflow the viewport on narrow devices.
auth.js:145 (@media max-width:768px)
Recommendation: Make .rz-user-dropdown full-width on mobile with position:fixed; bottom:0; left:0; right:0; as a bottom sheet pattern.
Medium Bento Grid Career Timeline May Overflow on Tablets
The career timeline row (.bento-row-career) renders 5 company cards horizontally. On tablet viewports (768–1024px) this row will require complex responsive breakpoints. Each card contains company logos, badges, and descriptions that need careful reflow.
index.html:274–331
Recommendation: Add a 1024px breakpoint that wraps the career cards into a 2x3 or scrollable horizontal layout with snap points.
Low Admin Panel Not Responsive
The admin panel (rz-ops-p7x3k9m.html) uses a fixed 260px sidebar (.sidebar{width:260px;position:fixed}) and .main-wrap{margin-left:260px}. No mobile breakpoints are present. While this is an internal tool, it is inaccessible on mobile devices.
rz-ops-p7x3k9m.html:22, 47
Recommendation: Add a collapsible sidebar with a hamburger toggle for the admin panel at breakpoints below 1024px.

4. Accessibility (WCAG 2.1 AA)

62
Current
→
82
Expected
Pass Skip Navigation Link Present
The homepage includes <a href="#main-content" class="skip-link">Skip to main content</a> with proper focus-visible styling. The skip link targets <main id="main-content">.
index.html:145, styles.css:6–22
Pass Images Have Descriptive Alt Text
All <img> tags in index.html have descriptive alt attributes. The hero photos include full alt text like "Bagus Dwi Permana - Engineering Operations Manager". Company logos also have alt text (e.g., "ZTE", "AWS"). 19 alt attributes were found across the homepage.
index.html (19 alt attributes found)
Pass Decorative SVGs Use aria-hidden
SVG icons in the metrics section correctly use aria-hidden="true", and article cards use aria-label attributes to provide context for screen readers.
index.html:463, 485, 509 etc.
High Auth Modal Lacks ARIA Dialog Attributes
The login modal (.rz-modal-overlay) does not use role="dialog", aria-modal="true", or aria-labelledby. Screen readers will not announce the modal context. Focus is not trapped within the modal — users can tab to elements behind the overlay.
auth.js:175–203 (loginModalHTML function)
Recommendation: Add role="dialog" and aria-modal="true" to .rz-modal. Add aria-labelledby="rzModalTitle" pointing to the h3. Implement focus trapping with a focus loop on first/last focusable elements.
Medium Color Contrast Issues in Secondary Text
Several text elements use colors with insufficient contrast against dark backgrounds: #64748b on #0f172a (ratio ~3.8:1, below 4.5:1 AA minimum), #94a3b8 meta text on dark cards (~4.3:1, borderline). The .bento-exp-desc and .rz-modal-sub elements are affected.
styles.css (--gray-600 / --slate-400 usage), auth.js:109
Recommendation: Increase secondary text color brightness to at minimum #9ca3af on dark backgrounds to achieve 4.5:1 contrast ratio.
Medium Keyboard Navigation Incomplete for Dropdown Menus
The navigation dropdown menus (.dropdown-menu) are triggered by CSS :hover only (.nav-dropdown:hover .dropdown-menu). There is no :focus-within support, meaning keyboard users cannot access DC Solutions or Insights sub-pages via the dropdown.
styles.css:324 (.nav-dropdown:hover only)
Recommendation: Add .nav-dropdown:focus-within .dropdown-menu { opacity:1; visibility:visible; } to enable keyboard access to dropdown items.
Medium No ARIA Roles on Major Landmark Sections
The page uses semantic HTML5 elements (<nav>, <main>, <section>) but no explicit role attributes were found. While HTML5 semantics provide implicit roles, the article pages with complex calculator interfaces would benefit from explicit ARIA landmarks for regions, complementary content, and form groups.
index.html (0 role= attributes found)
Recommendation: Add role="banner" to header, role="navigation" with aria-label to nav elements, and role="region" aria-label="..." to distinct calculator sections in articles.
Low Theme Toggle Button Accessible but Missing State
The theme toggle has aria-label="Toggle dark mode" which is good, but lacks aria-pressed to indicate current state. Screen reader users cannot tell if dark mode is currently active.
index.html:196
Recommendation: Add aria-pressed="false" to the button and toggle it to "true" when dark mode is active.

The CAPEX and OPEX calculators, as well as Pro Mode features (Monte Carlo simulations with 10K iterations, sensitivity analysis), perform heavy JavaScript operations. There is no visible loading spinner or progress indicator during computation, leading to a "frozen UI" perception.
Recommendation: Add a skeleton loader or progress bar for Monte Carlo simulations. Use requestAnimationFrame or Web Workers for heavy calculations to keep the UI responsive.

The #cursorSpotlight element follows mouse movement across the page. This requires continuous mousemove event handling and CSS transforms on every frame. On low-powered devices this could cause jank.
index.html:150
Recommendation: Throttle the mousemove handler to 60fps and disable the effect on mobile/touch devices and via prefers-reduced-motion media query.

6. Performance UX

74
Current
→
86

Pass Critical Resource Preloading
The homepage preloads the hero image (fetchpriority="high"), the CSS file, and fonts via <link rel="preload"> and <link rel="preconnect">. Article pages also preload cover images.
index.html:58–65, article-1.html:61
Pass Font Loading Strategy
Fonts use preload-as-style with onload="this.onload=null;this.rel='stylesheet'" pattern plus a <noscript> fallback. The display=swap parameter ensures text remains visible during font loading.
index.html:65–66
Medium Large Inline CSS in Article Pages
Each article page contains thousands of lines of inline CSS for Pro Mode calculators, charts, panels, and tooltips. Article-15 and Article-13 are particularly large. This increases initial HTML payload and time-to-interactive. The styles.css file alone is 5000+ lines.
Recommendation: Extract Pro Mode CSS into a separate pro-mode.css that is loaded conditionally (only when user has Pro session), reducing initial page weight for free users.
Medium Global CSS Transition on All Elements
The rule *, *::before, *::after { transition: background-color 0.3s, border-color 0.3s, box-shadow 0.3s; } applies transitions to every element in the DOM. This causes unnecessary style recalculations and compositor work, especially during theme switching on pages with hundreds of elements.
styles.css:147–149
Recommendation: Remove the universal transition rule. Apply transitions only to elements that actually change during theme switching: .navbar, .card, body, etc.
Low Lazy Loading Applied to Hero Avatar
The navbar avatar image uses loading="lazy" despite being above the fold: <img loading="lazy" src="assets/profile-photo.jpg">. This delays rendering of a visible element.
index.html:156
Recommendation: Remove loading="lazy" from the navbar avatar since it is visible on initial load. Only apply lazy loading to below-fold images.
